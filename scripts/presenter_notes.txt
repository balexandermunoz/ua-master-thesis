[Slide 3]
Modern cities depend on three interdependent infrastructure networks: energy, mobility, and telecommunications. They are not independent— electric vehicles link power grids and traffic, smart traffic needs telecom, and energy depends on communication networks for coordination.

The problem is that today's simulation tools work in isolation. Each one handles a single domain very well, but none of them can model interactions across domains. There are some platforms that are great at monitoring and aggregating data, but they can't answer predictive cross-domain questions — things like "what happens to the power grid when 500 EVs charge for one hour?" or "how does closing a road affect both traffic and the energy network?"

Our goal is to build a flexible, domain-agnostic simulation engine that supports multi-domain co-simulation and can answer exactly those kinds of what-if questions.


[Slide 4]
In the state of the art, we reviewed the main simulation tools in each domain. (Read and speak about the domains)
We also reviewed co-simulation frameworks: Mosaik is smart-grid focused, MECSYCO is DEVS-based, and HELICS was designed specifically for multi-domain infrastructure co-simulation with flexible time management. We chose HELICS. (bc mult domain, scalable, flexible time management = useful in the simulations and I’ll explain why later)


[Slide 5]
We adopted Design Science Research, which is convenient when the goal is to build and evaluate an engineering artifact rather than just observe a phenomena. The process has four phases. First, we identify the problem: existing simulation tools are domain-specific and there's no modular, reusable architecture for cross-domain urban simulation. Second, we design a three-layer architecture. Third, we implement the framework in Python with domain modules and concrete scenarios. Fourth, we validated the outputs against expected physical behaviors and known benchmarks. These phases are iterative — feedback to refinement.


[Slide 6]
Here you can see the overall architecture. It follows three layers. At the bottom, a domain-agnostic core engine handles all generic simulation concerns. In the middle, independent domain modules encapsulate the logic for energy, mobility, and telecom. At the top, HELICS orchestrates cross-domain co-simulation. The key principle is strict separation: the engine knows nothing about power grids or vehicles, and domain modules know nothing about each other. They only communicate through well-defined interfaces and HELICS publish-subscribe topics.



[Slide 7]
The core engine provides four functionalities. Agents are autonomous entities with a step function called every tick. Time control supports configurable resolution from 100 milliseconds up to 15 minutes. The engine defines a contract: every domain module must implement initialize, step, and generate_report. And a generic collector accumulates metrics without knowing what they represent. This means adding a new domain — say water or waste management — requires no changes to the engine. You just implement the contract and plug in.


[Slide 8]
We built three domain modules. The energy module models distribution networks with solar panels, wind turbines, battery storage, and demand-responsive loads. The mobility module is an agent-based traffic simulation with A-star routing and adaptive signal control. The telecom module simulates 5G network slicing with resource allocation across multiple stations. Crucially, each module runs identically in standalone mode or as a HELICS federate.
Mathematics is the language used to abstract the domain concepts into agents, generating realistic data and enabling different domains to communicate with each other.


[Slide 9]
Helics supports both time-stepped and event-driven federates, which is essential because our energy module runs at 15-minute steps while telecom runs at 100-millisecond events. HELICS manages federate registration, time synchronization, and data exchange through value federates. And adding a new domain means registering new topics, not modifying existing modules.



[Slide 10]
We implemented five scenarios. E1 and E2 cover the energy domain — smart grid with renewables and EV charging infrastructure. M1 covers urban traffic. T1 covers 5G network slicing. And E2 plus M1 is our cross-domain scenario coupling energy and mobility. Time steps range from 100 milliseconds in telecom to 15 minutes in the grid scenario. All scenarios support both standalone and co-simulation modes.


[Slide 11]
Evaluation has two dimensions. For domain correctness, each scenario has specific physical expectations — voltages within IEEE bounds, smart charging reducing peak demand, adaptive signals reducing travel time, dynamic slicing improving utilization, and coupling producing measurably different outcomes than isolation. For architectural validation, we verify that modules behave identically in standalone and co-simulation modes, that all agents comply with the engine contract, that strategy variants produce statistically distinct results, and that HELICS time grants ensure proper synchronization.


[Slide 12]
This slide summarizes the development environment used to build and reproduce the framework.
The implementation is fully in Python 3.11, developed on Windows 11 with Visual Studio Code.
For the technical stack, HELICS provides the co-simulation middleware, NumPy supports numerical operations, and Streamlit powers the web interface for running scenarios.
Version control is handled with Git and GitHub, and dependency management is intentionally simple through requirements.txt.

From a reproducibility perspective, setup is straightforward: create the virtual environment, install dependencies, and launch the dashboard with streamlit run code/app.py.
So the key point here is not only what tools were used, but that the full environment can be recreated quickly by any reviewer with standard Python tooling.

[Slide 13]
Here I show how the codebase maps directly to the three-layer architecture proposed in the methodology.
Layer 1 is the core engine in engine/base.py, where the BaseFederate abstraction handles common simulation concerns like orchestration, time stepping, and HELICS integration points.
Layer 2 is the scenarios package, with one module per scenario, including the domain-specific and cross-domain cases.
Layer 3 is app.py, which provides the Streamlit interface and acts as the orchestration entry point for execution and comparison.

This structure enforces separation of concerns: generic logic stays in the engine, domain behavior stays in scenario modules, and user interaction stays in the UI layer.
That separation is what makes the framework easier to maintain, extend, and validate.

[Slide 14]
This table gives a compact implementation snapshot of all scenarios.
The first key point is scale: the number of agents ranges from 125 in E2 up to more than 2,500 in the mobility and cross-domain cases.
The second point is time resolution: each scenario keeps the time step that best matches its domain physics, from 100 milliseconds in telecom to 15 minutes in the smart-grid case.
The third point is strategy flexibility: several scenarios support two or three strategy variants, which is important for controlled comparisons.

So this slide is not only descriptive; it shows that the framework can handle heterogeneous scales, different temporal granularities, and multiple decision strategies under one common architecture.

[Slide 15]
This slide focuses on BaseFederate, the core abstraction of the engine.
Conceptually, it defines a clean contract with three abstract methods: initialize components, run the simulation loop, and generate the report.
Every domain module implements that same contract, which gives us consistency across energy, mobility, telecom, and cross-domain scenarios.

At the same time, BaseFederate provides reusable concrete behavior, especially HELICS setup and time advancement.
That means domain code does not need to include co-simulation boilerplate.
Another practical benefit is dual-mode execution: the same module can run standalone or with HELICS by changing a flag, not by rewriting logic.
This is one of the main reasons the framework remains modular and easy to extend.

[Slide 16]
Here we move from architecture to usability.
The Streamlit interface is the entry point for running all scenarios without writing code.
Users select the scenario, adjust parameters, choose strategy or coupling mode, and execute directly from the browser.

A key design decision was to support both single-run reports and compare mode in the same workflow.
Results are displayed in structured metric sections, and comparison runs are shown side by side so trade-offs become visible quickly.
We also keep outputs in session state to avoid unnecessary recomputation when users interact with widgets.
So this layer is not just a demo front-end; it is what makes the framework practical for repeatable experimentation by non-developers.

[Slide 17]
This slide summarizes the main technical challenges we had to solve during implementation.
First, HELICS dependency isolation: we needed graceful fallback when native libraries are unavailable, so standalone execution still works.
Second, scalability in mobility loops: with millions of agent steps, we optimized where possible, for example by caching route computation.

Third, cross-domain coordinate alignment: different modules use different spatial representations, so we introduced mapping layers to couple them consistently.
Fourth, heterogeneous time steps: scenarios span several orders of magnitude in temporal resolution, so synchronization had to remain explicit and robust.

Overall, these challenges were not incidental bugs; they were architectural tests, and solving them strengthened the framework's portability and extensibility.

So this is the point where I can say the implementation is not just working structurally, but also producing sensible outputs for each individual domain.
The detailed tables later in the appendix show the exact values, but here I want the audience to keep the overall pattern in mind: each scenario behaves as expected and the strategy choices produce meaningful differences.

[Slide 19]

Second, it supports multiple domains and cross-domain integrations in a way that keeps the code reusable and modular.
Third, it makes the trade-offs visible through strategy substitution, which is important for comparing alternatives fairly.

[Slide 21]
This slide is about the limitations, and they are important because they define the boundary of the prototype.
In the energy case, the voltage model is simplified, so it does not yet capture full AC power-flow behavior.
In mobility, routes are computed once and not continuously rerouted, which keeps the model manageable but less realistic.
And in telecom, we still use an abstracted radio model instead of a full packet-level network simulation.
These are acceptable for this dissertation because the goal was architectural validation, but they are the first places I would improve in future work.

[Slide 22]
This slide points to the next steps.
The most immediate improvement is to replace the simplified energy model with a proper AC power-flow solver.
After that, I would extend the framework to a full multi-process HELICS deployment so the coupling runs exactly as it would in a distributed setting.
I would also add dynamic rerouting in mobility and richer wireless effects in telecom to increase realism.
So the future work is very clear: keep the same architecture, but progressively increase fidelity and scale.

So the contribution is both architectural and practical, because it gives a reusable way to study complex urban systems.

[Slide 30]
Scenario E1 models an IEEE 33-bus distribution network with 50 solar panels, 3 wind turbines, 
5 battery units, and 800 residential loads with demand response. We validate voltage profiles, 
curtailment behavior, and DR load reduction. Scenario E2 puts 100 EVs on a 13-node feeder and compares 
three charging strategies: uncoordinated, where everyone charges immediately; smart, 
which prioritizes by urgency — energy needed divided by time available; and V2G, which adds bidirectional 
power flow during peak hours at a 20% premium. We use the strategy pattern so all three variants share 
the same simulation loop with no code duplication.

[Slide 31]
M1 simulates 2,500 vehicles on a 5-by-5 kilometer grid with 25 signalized intersections. Vehicles route using A-star with a Manhattan heuristic plus random perturbation for route diversity. We compare fixed-time signals, which give equal green to both directions, against adaptive signals, which allocate green time proportionally to the measured queue at each phase change. The formula is simple: green time scales linearly between 15 and 90 seconds based on the ratio of directional demand. We also track CO2 emissions with a two-state model — idle emissions at 2.3 grams per second and moving emissions at 0.15 grams per meter.

[Slide 32]
T1 simulates three 5G base stations serving 200 mobile users across three network slices: eMBB for broadband, URLLC for low-latency, and mMTC for IoT devices. We use the 3GPP path loss model with shadow fading, random waypoint mobility, and a handover mechanism with hysteresis and cooldown to prevent ping-pong effects. We compare static slicing, which fixes resource block allocation at 50-30-20 percent, against dynamic slicing, which redistributes proportionally to instantaneous demand. This is conceptually analogous to M1's adaptive traffic signals — both allocate capacity based on real-time demand rather than fixed assumptions.

[Slide 33]
This is the key integration scenario. Five hundred EVs share the road network with 2,000 background vehicles. Thirteen charging stations provide 52 ports. We compare two modes under the same random seed. In coupled mode, EVs physically drive through traffic, wait at red lights, and cause congestion. In uncoupled mode, EVs teleport to their station — treating energy and mobility as independent domains. The multi-rate alignment runs traffic at 1-second ticks and charging logic every 5 minutes. We measure three coupling-specific metrics: average EV drive time, which should be zero uncoupled but significant coupled; emission uplift from EV driving; and SOC target deficit, since travel delays reduce available charging time.
